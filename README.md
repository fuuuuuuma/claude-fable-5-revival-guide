## 1. 3分でわかる「Fable 5 復活」の全体像

まず、今回のニュースを時系列でひとことずつ。

- **6/9**: Anthropic が新モデル **Claude Fable 5 / Claude Mythos 5** を発表 ✅
- **6/12**: 米政府が「national security authorities（国家安全保障上の権限）」を根拠に輸出管理指令を発出。Anthropic は Fable 5 / Mythos 5 への**全ユーザーのアクセスを停止** ✅
- **6/30**: 輸出規制が**解除** ✅
- **7/1**: Fable 5 が**全世界のユーザーに復活**。あわせて 7/7 までのプロモーションアクセス（定額プランの週間上限の最大50%まで追加費用なし）が始まった ✅

つまり「復活」とは、単なる新機能の追加ではなく、**約19日間使えなかった最上位モデルが、規制解除とともに戻ってきた**という出来事です。X 上でも「復活してた🎉」「We are so back」といった反応が並び、復活初日から Claude Code で長時間タスクを走らせる報告が増えています 🔶（[復活を喜ぶ反応の例](https://x.com/TOMOKIN_Voice/status/2072163869095809076)）。

そして本資料の後半では、「戻ってきた高性能モデルに、いきなり大仕事を任せる前にやっておきたい2つの準備」を扱います。

1. **Skills・CLAUDE.md まわりの一気修復** — 約19日間、別モデルで運用していた間に溜まった記憶・設定のズレやノイズを、整理プロンプト（memory-dream 型）で一掃する
2. **ハーネス構築** — Claude Code 公式ブログ3本（Steering Claude Code / Getting started with loops / Building effective human-agent teams）のベストプラクティスに沿って、「指示の置き場所」と「任せ方」を設計し直す

「ハーネス（harness）」は馬具の「手綱」のこと。AI の世界では、**モデルの周りに置く道具立て一式**（CLAUDE.md・スキル・フック・検証の仕組みなど）を指します。モデルが強くなるほど、手綱側の設計が成果を左右します。

## 2. Fable 5 とは — Opus の上に置かれた「Mythosクラス」

Claude Fable 5 は、Anthropic の新しい **Claude 5 ファミリー最初のモデル**で、これまで最上位だった Opus の**さらに上**に位置づけられた新階層「**Mythosクラス**」に属します ✅。

ポイントは、**Fable 5 と Mythos 5 は同じ基盤モデル**だということ。違いは安全対策の層です ✅。

| | Fable 5 | Mythos 5 |
|---|---|---|
| 基盤モデル | 同一 | 同一 |
| 安全対策 | dual-use（軍民両用）能力向けの追加セーフガードあり | 追加セーフガードの一部を解除 |
| 使える人 | 一般提供 | 承認済み組織のみ（サイバー防御の Project Glasswing パートナー、生物学の信頼アクセスプログラム参加研究者など） |

ひとことで言えば、**Fable 5 は「一般の私たちが使える形に安全装置を付けた Mythos 5」**です。日常の開発・執筆・分析でこの差を意識する場面はほとんどなく、私たちにとっての「最上位モデル」は実質 Fable 5 になります。

公式発表では「ほぼすべてのテスト済みベンチマークで最先端であり、**タスクが長く複雑なほど Fable 5 の優位性が顕著**」と説明されています ✅。詳しい実力は [5章](#5-fable-5-の実力と価格) で。

## 3. 何が起きていたのか — 停止から復活までのタイムライン

### 3-1. 停止：6月12日

2026年6月12日（米東部時間 17:21）、米政府は「national security authorities」を根拠に、Fable 5 と Mythos 5 を対象とする輸出管理指令を発出しました ✅。指令の対象は「米国内外を問わず、外国籍の全ユーザー」——Anthropic の外国籍従業員も含む広い範囲でした ✅。

Anthropic は「ユーザーの国籍をリアルタイムで確認する手段がない」ため、**両モデルへの全ユーザーアクセスを一時停止**する対応を取りました ✅。対象は Fable 5 / Mythos 5 のみで、Opus 4.8 など他のモデルは影響を受けていません ✅。

### 3-2. きっかけ：脆弱性発見のデモ

報道と公式ページによると、きっかけは **Amazon の研究者が発見した「Fable 5 のセーフガードを回避する手法（jailbreak）」の報告**でした ✅。プロンプトの工夫でモデルにソフトウェアの欠陥を特定させ、一部ではその欠陥を悪用するコードを書かせられた、という内容です 🔶（[The Hacker News の報道](https://thehackernews.com/2026/07/anthropic-restores-claude-fable-5-after.html)）。

一方で Anthropic は公式声明で異議も表明しています ✅。

- 政府が提示したのは「比較的単純な既知の脆弱性」を示すデモだった
- Anthropic の検証では、**Opus 4.8・GPT-5.5・Kimi K2.7 などのより低性能なモデルでも同じ脆弱性を特定でき**、報告されたエクスプロイト実証はテストした全モデル（Haiku 4.5 から Opus 4.8 まで）が同等の出力を生成した
- 「狭い範囲の jailbreak の発見が、数億人に展開された商用モデルの回収理由となるべきではない」

### 3-3. 解除と復活：6月30日 → 7月1日

6月30日に輸出規制は解除され ✅、Fable 5 は **7月1日から全世界のユーザーに提供再開**されました ✅。復活にあわせて Anthropic は、Amazon の報告にあった特定の回避手法を検出する**改良版の安全分類器**を導入。公式ページによると「その特定技術を99%以上のケースでブロック」し、ブロックされたリクエストは Opus 4.8 に振り替えられます ✅。

停止期間は 6/12〜7/1 の**約19日間**でした ✅。

普段のコーディングや資料作成でフォールバックに当たることはまれですが、「あれ、いつもより返答の雰囲気が違う？」と感じたときは、この仕組みを思い出してください。

## 7. 復活したらまずやること① Skills・CLAUDE.md の一気修復（memory-dream）

### 7-1. なぜ「修復」が先なのか

Fable 5 が止まっていた約19日間、多くの人は Opus 4.8 など別のモデルで作業を続けていました。その間にも CLAUDE.md・Skills・メモリには新しい追記が積み重なっています。長く運用した設定ファイルには、次のようなノイズが溜まりがちです。

- 「明日やる」「先週の件」のような**相対日付**が、いつの話か分からなくなっている
- すでに削除した手順・ツールへの**参照の残骸**
- 上位ファイルと下位ファイルで**同じルールの重複**や、古い記述と新しい記述の**矛盾**

高性能なモデルほど、渡された指示を忠実に読み込みます。つまり**ノイズ入りの設定を渡すと、ノイズまで忠実に実行してしまう**。Fable 5 のような最上位モデルに切り替えるタイミングこそ、設定の大掃除に最適です。

### 7-2. memory-dream — X で共有された整理プレイブック

この「一気修復」の具体的な手順として参考になるのが、conao_3 氏が X で公開した「**memory-dream**」というプレイブックです 🔶（[元投稿](https://x.com/conao_3/status/2065815666310238330)）。Anthropic の自動記憶整理（auto-dream）の考え方を、git ベースの手動環境で再現するもので、要点は次の4フェーズです（以下は元投稿をもとにした要約・再構成 🔶）。

1. **Mine（採掘）**: 直近のセッション記録から、繰り返し出てきた指摘・確定した方針・新しい事実を抽出する。一回限りのデバッグメモは無視する
2. **Consolidate（統合）**: 抽出したものを既存の記憶ファイルにマージする。相対日付は絶対日付に変換し、矛盾があれば最新の情報で上書き、存在しないファイルへの参照は削除する
3. **Dedup & Resolve（重複排除・最重要）**: 上位レイヤ（CLAUDE.md 等）に書いたルールは、下位のファイルで再掲しない。重複を見つけたら下位側を削除する。「なぜ消したか」のメタ説明も残さない
4. **Prune & Index（剪定・索引化）**: 索引ファイル（MEMORY.md 等）を200行以内に抑え、ノート一覧を実ファイルから再生成する

そして、成果ファイルの書き方の鉄則がこれです。

> 判断の経緯・失敗談・「N回指摘された」などの履歴は書かない。残すのは**現時点で正しい知見・ルール・再現手順のみ**。

### 7-3. そのまま使える修復プロンプト

元投稿の使い方に沿った実行指示はこの形です（Skills として保存してから呼び出す運用が推奨されています 🔶）。

```
memory-dream を実行して、現在の記憶階層全体（CLAUDE.md / Skills / メモリファイル）を整理してください。

手順:
1. Mine: 直近のセッション記録から、繰り返し出た指摘・確定方針・新事実だけを抽出（1回限りのメモは無視）
2. Consolidate: 既存ファイルへマージ。相対日付→絶対日付、矛盾は最新で上書き、死んだ参照は削除
3. Dedup & Resolve: 上位ファイルにあるルールを下位から削除。メタ説明は残さない
4. Prune & Index: 索引は200行以内に。ノート一覧を実ファイルから再生成

鉄則: 経緯・失敗談・履歴は書かない。現時点で正しい知見・ルール・再現手順のみ残す。
出力はレビューできる形で、変更をコミットとして提示してください。
```

これを Fable 5 に投げると、19日分（あるいはもっと長期間）の設定のズレを一度で洗い出せます。**出力を必ず自分でレビューしてから確定する**のがポイントです（勝手に確定させない）。

### 8-1. 指示の「置き場所」は7つある（Steering Claude Code）

Claude Code に指示を与える方法は、CLAUDE.md だけではありません。公式は7つの方法を、「いつ読み込まれるか」「長いセッションで保持されるか」「どれだけ強制力があるか」で使い分けよ、と整理しています ✅。

| 置き場所 | 向いているもの |
|---|---|
| CLAUDE.md（ルート） | ビルドコマンド・構成・チームの慣習など「常に必要な事実」。**200行未満に保つ** |
| CLAUDE.md（サブディレクトリ） | そのフォルダ固有の規約（触れたときだけ読み込まれる） |
| ルール（.claude/rules/） | 「API ハンドラは必ず入力検証」など特定の制約。**paths 指定**で関係あるときだけ読み込む |
| スキル（.claude/skills/） | デプロイ手順・チェックリストなど**手続き**。呼び出し時だけ本文が読まれる |
| サブエージェント | 深い調査・ログ解析など、隔離して要約だけ受け取りたい副次タスク |
| フック | リンター実行・完了時通知・危険コマンドのブロックなど、**確実に起きてほしい自動化** |
| 出力スタイル / システムプロンプト追記 | 役割やトーンの大きな変更（慎重に） |

特に重要なのが、公式が挙げる「**指示の置き場所を見直すべきサイン**」です ✅。

- 「毎回 X のときは必ず Y」を CLAUDE.md に書いている → それは**フック**の仕事（モデルの判断に頼らず自動実行）
- 「これは絶対にしてはいけない」を CLAUDE.md に書いている → 本当のガードレールは指示では作れない。**フック（PreToolUse で拒否）と権限設定**で強制する
- CLAUDE.md に30行の手順を書いている → 手順は**スキル**へ
- 特定のフォルダにしか関係ないルールに paths を付けていない → **paths 指定**で絞る
- 個人の好みをプロジェクトの CLAUDE.md に書いている → **ユーザーレベルのファイル**へ

### 8-2. ループには4種類ある（Getting started with loops）

公式ブログは、ループを「**決められた停止条件を満たすまで、エージェントが作業のサイクルを繰り返すこと**」と定義し、4種類に整理しています ✅。

1. **① ターン型**: プロンプトを送るたびに1サイクル走る基本形。普段の対話がこれ
2. **② ゴール型（/goal）**: 「完了の定義」を先に与え、達成か試行上限まで自走させる。テスト通過数など機械的に判定できる基準が最も効く
3. **③ 時間型（/loop・/schedule）**: 指定間隔で繰り返す。PR 監視や定期チェック向き
4. **④ プロアクティブ型**: イベントやスケジュールを起点に、人間がリアルタイム関与せず動き続ける構成

本資料で扱う「ハーネス構築プロンプト」（次章）は、このうち **①ターン型**として1回実行するタスクです。そして構築したハーネス（検証スキル・フック・明確な完了条件）が、後々 ②〜④ の自律的なループを安全に回すための土台になります。

記事の核メッセージは2つ。「**判定基準が数値化されているほど、Claude は自己検証しやすい**」「**まずは一番シンプルなやり方から始める**」 ✅。

### 8-3. チームとして運用する（Building effective human-agent teams）

3本目の記事は、エージェントを「同僚」として運用するための教訓集です ✅。ハーネス構築の観点で効くのは次の3点。

- **検証手段を複数持たせる**: 最も優れた長時間エージェントは、人間が確認する前に自分の作業を検証する方法を複数持つ。コードならテスト、文書ならルーブリック（評価基準）
- **実行者と検証者を分ける（Doer-Verifier）**: 実装した本人は自分の思考に引きずられる。まっさらな文脈のレビュー役エージェントを別に立てると偏りが減る
- **信頼は段階的に広げる**: 最初は人間が全部レビューし、うまくいったタスク種別から自律性を広げる。あるチームは最終的に500件のバグ修正を独立して任せられるようになった

## 9. ハーネス構築用プロンプト（コピペ用）

以上の公式ベストプラクティスを1本のプロンプトに落とし込むと、こうなります。Fable 5 復活後の「2手目」として、自分のリポジトリで実行してみてください（①ターン型として1回走らせるだけです）。

```
このリポジトリの Claude Code 設定（ハーネス）を、公式ベストプラクティスに沿って診断し、
再設計案を提示してください。基準は以下です。

【置き場所の基準】
1. CLAUDE.md（ルート）は「常に必要な事実」（ビルドコマンド・構成・規約）だけに絞り、200行未満に保つ
2. 「毎回Xのときは必ずY」型の指示は、CLAUDE.md から settings.json のフックへ移す
3. 「絶対にしてはいけない」型の禁止は、フック（PreToolUse で拒否）と権限設定で強制する
4. 30行を超える手順は .claude/skills/ のスキルへ移す
5. 特定パスにしか関係ない規約は、paths 指定つきのルール（.claude/rules/）へ移す
6. 個人の好みはユーザーレベルの設定ファイルへ移す
7. 深い調査・ログ解析のような隔離できる副次タスクは、サブエージェントとして定義する

【ループの基準】
8. 繰り返し実行するタスクには「完了の定義」と「試行回数の上限」を明文化する
9. 成果物の自己検証手順（テスト・チェックリスト・ルーブリック）をスキルとして書き出す
10. 実装役とは別に、レビュー役（検証者）のエージェント運用を用意する

手順: まず現状の CLAUDE.md / rules / skills / hooks / agents を読み、
「どの指示を・どこへ・なぜ動かすか」の一覧表を出してください。
私が承認したら、実際の移行を1項目ずつ実行してください。
```

このプロンプトのいいところは、**いきなり書き換えず「診断→一覧表→承認→移行」の順にする**点です。ハーネスは一度壊すと復旧が大変なので、人間のレビューを挟む設計にしています。

## 11. もう一歩進んだ使い方

開発寄りの人向けに、少しだけ発展的な話を。

- **50%枠の配分**: プロモーション枠は週間上限の50%まで ✅。日常の軽い往復は Opus 4.8 / Sonnet に任せ、Fable 5 は「リポジトリ横断のリファクタ」「長時間の自律タスク」「大量ドキュメントの統合分析」に温存すると効率的です
- **/goal との相性**: 完了条件が数値で書けるタスク（テスト全通過、Lighthouse 90点以上など）は、②ゴール型ループで Fable 5 に自走させると、長時間タスク耐性が活きます
- **使用量の確認**: `/usage` でスキル・サブエージェント・MCP ごとの使用量を、引数なしの `/goal` でターン数とトークン使用量を確認できます ✅（公式ブログ記載の運用術）
- **Doer-Verifier 構成**: Fable 5 を実装役に、別エージェントを検証役に置く二重化は、公式が推す品質担保パターンです ✅

**Q. 無料プランでも Fable 5 を使えますか？**
A. 今回のプロモーションアクセスの対象は Pro / Max / Team / 一部 Enterprise で、無料プランは対象外です ✅。

**Q. 7/7 を過ぎたらどうなりますか？**
