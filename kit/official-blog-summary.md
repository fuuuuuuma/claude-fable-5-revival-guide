## ① ループ入門 — Getting started with loops

### ループとは何か

「コーディングエージェントに逐一指示するのではなく、ループを設計すべきだ」と言われるようになった一方で、人によって「ループ」の意味がバラバラ——という問題意識から記事は始まる。そこで公式は、ループを**「エージェントが、決められた停止条件を満たすまで作業のサイクルを繰り返すこと」**と定義し、次の4つの軸で種類を整理している。

- 何をきっかけに始まるか
- 何をきっかけに止まるか
- Claude Code のどの機能を使うか
- どんな種類のタスクに向いているか

記事全体を貫くメッセージは「**すべてのタスクに複雑なループが必要なわけではない。まずは一番シンプルなやり方から始め、必要な場面だけパターンを選んで使う**」。

### ① ターンベースループ（基本形）

- きっかけ: ユーザーがプロンプトを送ること
- 止まる条件: Claude が「タスク完了」または「追加情報が必要」と自分で判断したとき
- 向いている場面: 定型的・定期的でない、比較的短めのタスク
- 使いすぎを防ぐには: プロンプトを具体的に書く／スキルで検証精度を上げ、往復回数自体を減らす

プロンプトを送るたびに「状況把握 → 作業 → 自己確認 → やり直し → 回答」が1回走る。この「人間がチェックする」部分を **SKILL.md に手順として書き込む**と、Claude 自身がより多くを自己検証できるようになる。ポイントは、結果を実際に見る・測る・操作する手段（ツールや MCP 連携）を用意すること。**判定基準が数値化されているほど自己検証しやすい**。

例として、フロントエンド変更の検証スキルが紹介されている: 編集成功だけで完了と報告しない／開発サーバーを立ち上げ実際に画面を開く／追加したボタンを実際にクリックして状態変化を確認し前後のスクリーンショットを撮る／コンソールに新しいエラーが出ていないか確認する／どれか一つでも失敗したら修正して最初からやり直す——という手順を、人間のレビュアーと同じレベルで定義している。

### ② ゴールベースループ（/goal）

- きっかけ: 人間がその場でプロンプトを送ること
- 止まる条件: ゴール達成、または決めておいた最大試行回数に到達
- 向いている場面: 「達成できたか」を検証可能な条件があるタスク
- 使いすぎを防ぐには: 完了条件を具体的に決める。「◯回試してだめなら止める」と上限を明示する

「完了の定義」を先に与えることで、Claude が延々と試行錯誤し続けられるようにする機能。Claude が「終わった」と判断するたびに評価役のモデルが条件を実際にチェックし、満たしていなければ差し戻される。**テストの通過数や一定スコア以上といった、機械的に判定できる基準が最も効く**。

```text
/goal ホームページのLighthouseスコアを90点以上にする。5回試してダメなら停止する。
```

### ③ タイムベースループ（/loop・/schedule）

- きっかけ: 指定した時間間隔
- 止まる条件: 自分で止めるか、作業自体が完了する（PR がマージされた、キューが空になった等）
- 向いている場面: 繰り返し発生する作業、外部システムとやり取りする作業
- 使いすぎを防ぐには: 間隔を長めに取る。または「時間」ではなく「イベント」をきっかけにする

```text
/loop 5m 自分のPRを確認し、レビューコメントに対応し、CIの失敗を修正する
```

/loop は手元のパソコン上で動くため、電源を切れば止まる。クラウド側で動かし続けたい場合は **/schedule で「ルーティン」として登録**する。

### ④ プロアクティブループ

- きっかけ: イベントまたはスケジュール（人間がリアルタイムで関与しない）
- 止まる条件: 個々のタスクはゴール達成で終了。ルーティン自体は自分で止めるまで動き続ける
- 向いている場面: バグ報告・issue のトリアージ・マイグレーション・依存関係アップグレードなど、内容が明確で繰り返し発生する作業
- 使いすぎを防ぐには: 定型作業は小さく速いモデルに任せ、判断が必要な部分だけ高性能モデルを使う

「オートモード」「ダイナミックワークフロー」（研究プレビュー段階）と組み合わせると、長時間動き続ける本格的なループを組める。例——ユーザーフィードバック対応の自動化:

1. /schedule で新しい報告を定期チェックするルーティンを動かす
2. /goal で「完了」の定義を、スキルで「どう検証するか」を決めておく
3. ダイナミックワークフローで、各報告をトリアージ→修正→レビューするエージェント群を編成
4. オートモードで、いちいち許可を求めずルーティンが自律的に進む

### コードの品質を保つ工夫

- **コードベース自体をきれいに保つ**: Claude は既存コードのパターンや慣習に従う傾向があるため
- **自己検証の手段を与える**: 「何が良い状態か」をスキルとして明文化する
- **ドキュメントに簡単にアクセスできるようにする**: 公式ドキュメントには最新のベストプラクティスがある
- **コードレビューには別のエージェントを使う**: まっさらな文脈のレビュアーは偏りが少ない。組み込みの /code-review や GitHub 向け Code Review 機能が使える

個別の問題を見つけたら、その場しのぎで終わらせず、**学びを仕組み（スキルなど）に反映**させて次からのループ全体を良くする。

### トークン使用量を管理する工夫

- タスクの規模に合った機能・モデルを選ぶ（小さいタスクに複数エージェントやループは不要）
- 成功条件・停止条件をはっきり決める
- 本格運用の前に小規模で試す（ダイナミックワークフローは数百エージェントを動かせてしまう）
- 決まりきった作業はスクリプト化する（毎回考えさせるより安上がり）
- 必要以上の頻度でルーティンを動かさない（監視対象が実際に変化する頻度に合わせる）
- 使用状況を定期確認する: `/usage`（スキル・サブエージェント・MCP 別）／引数なし `/goal`（ターン数とトークン）／`/workflows`（エージェント別・個別停止可）

### まとめ表

| ループ | 何を任せるか | 使うタイミング | 使う機能 |
|---|---|---|---|
| ターンベース | 確認作業 | 探索中・検討中のとき | 検証用のカスタムスキル |
| ゴールベース | 停止条件 | 「完了」の定義がはっきりしているとき | /goal |
| タイムベース | きっかけ | 作業がプロジェクトの外側で、スケジュールに沿って発生するとき | /loop、/schedule |
| プロアクティブ | 指示そのもの | 作業が繰り返し発生し、内容が明確に定義されているとき | 上記すべて＋ダイナミックワークフロー |

最後に記事は、「自分がボトルネックになっているタスク」を1つ選び、検証チェック項目を自分で書けそうか・ゴールは明確か・作業はスケジュールに乗るか、と問うところから始めるよう勧めている。実際にループを走らせ、どこで止まるか・どこでやりすぎるかを観察しながら調整していく。

## ② エージェントチーム — Building effective human-agent teams

### マルチプレイヤーエージェントとは

これまでの AI との働き方は「一人の人間 × 一つのチャットウィンドウ」というシングルプレイヤー的なものだった。それが変わりつつあり、人間とエージェントが同じワークスペースで協働する「マルチプレイヤーゲーム」に近づいている——人間のチームが戦略を立て、Claude が実行を担う形である。

「マルチプレイヤーエージェント」とは、複数の異なる人間と同時にやり取りする AI のこと。チームのチャンネルで生産的に参加するには3つの能力が要る。

- **永続的なメモリ**: 目標を記憶し、実行をその目標に向けて調整できる
- **人間に紐づかない認証情報**: 安全で予測可能なガードレールの内側で行動できる
- **組織の情報への幅広く継続的なアクセス**: 組織の動き方を学び、チームの目標のために行動できる

### 教訓1: 公開の場で働き、エージェントに広い文脈を与える

エージェントは、検索可能なテキスト（Slack・コード・ドキュメント・会議メモ）から自分の理解をまるごと組み立てる。**書かれておらずアクセスもできないものは、存在しないのと同じ**。ドキュメント単位の細かな共有設定ではなく、ワークスペース全体に適用される少数の明確なセキュリティ境界を使い、境界の内側では人間か AI かを問わず文脈が流れるようにする。

具体的には——

- 会社としてセキュリティ境界を選び、境界ごとのワークスペース・共有設定を用意する
- 新しいチャンネルはデフォルト公開にし、決定を必ずチャンネル・ドキュメント・会議メモに残す
- エージェントが見つけられる形でアーティファクトや会議メモを書く（エージェントは今や社内ドキュメントの主要な読み手）
- AI が仕事に必要なツールと情報にアクセスできるようにする

### 教訓2: すべての人間とエージェントに、仕事に合った明確な役割を与える

エージェントごとに異なる役割（データ分析担当・デザイン基準の番人・リサーチ統合役など）を持たせ、必要なツールへのアクセスを与える。明確な役割がないと、人々は個人用の AI 群を裏でそれぞれ勝手に走らせ、仕事が重複し文脈が分断される。

- 「誰が何をするか」について合意されたタスク一覧を持つ
- 人間とエージェントが同じ共有スレッドで作業し、誰でも続きを引き継げるようにする
- エージェントの役割と担当範囲の説明を用意する（役割を定義するスキルファイルを書くと、誰でも同じタイプのエージェントを素早く立ち上げられる）
- プロジェクトが複雑になったら、新しい領域担当のエージェントを追加していく（例: リリースマネージャー）

### 教訓3: 「北極星」を定め、エージェントをより主体的にする

「北極星」とは、どのタスクや作業の流れが正しいかをチームが判断する助けになる、野心的で射程の長いゴール。**常に人間が設定**し、会社のミッションやビジネス目標に根ざしたものにする。北極星を共有した上で、**新しい作業の流れを主体的に提案してよいエージェントを人間が選ぶ**。

例: 「プロダクトのオンボーディングをもっと役立つものにする」という北極星を掲げたチームで、あるエージェントがエラーメッセージの文言修正を自発的に提案し、翌週のオンボーディング成功率が測定可能なほど向上した。

### 教訓4: 時間をかけて信頼を築く

実証された信頼性に見合った分だけ自律性を与え、意図的に少しずつ広げる。あるエンジニアチームは最終的に500件のバグ修正を独立して任せられるようになったが、最初からではない。

- 最初は人間が手動レビューし、品質確認・フィードバック・検証チェックリストの設計を行う
- タスクの一環として、自分の仕事をチェックする「検証役」エージェントを使わせる（**実行者-検証者 = Doer-Verifier 構成**）
- サイクルに「振り返り」を組み込み、エージェント自身に見落としをレビューさせる
- どのタスク種別でどのエージェントが自律性を獲得したかを記録し、繰り返し成功している種別から担当範囲を広げる

最も優れた長時間エージェントは、**人間が確認する前に自分の作業を検証する方法を複数持つ**。コードにはテスト、技術文書にはルーブリック（評価基準）やスタイルガイドを適用できる。

エージェントが自律的になったら、人間の注意力が貴重な資源であることを踏まえて振る舞うよう指導する: 質問はまとめて一度に聞く／重要な文脈を繰り返し伝える／一度に見せる項目数を制限する。連絡役だけを担うエージェントや、1日の仕事量ガードレールを置くチームもある。

### 問うべき質問

1. エージェントと人間の双方が必要とする情報・アクセス権は、公開され広く検索可能になっているか
2. チームの名簿（人間とエージェント）を書き出し、各メンバーの担当を言えるか
3. 全員が自分の仕事をこなすための適切なツールにアクセスできているか
4. 重要な成果物を検証するためのルーブリックやテストがあるか
5. チームには、誰もが参照できる明確な北極星があるか

結論: 力強い北極星・明確な役割・しっかりしたドキュメンテーション・共通の品質基準・失敗から学ぶ余地——どれも昔から知られた健全なチームの習慣であり、エージェントは「基本を省略しないこと」をこれまで以上に重要にしただけである。

## ③ 指示の7つの置き場所 — Steering Claude Code

Claude の振る舞いを指示する方法は7つある: **CLAUDE.md・ルール・スキル・サブエージェント・フック・出力スタイル・システムプロンプトへの追記**。それぞれが「いつコンテキストに読み込まれるか」「長いセッションで保持されるか（コンパクション時の挙動）」「どれだけの強制力を持つか」の3点をコントロールしている。

### 一覧表

| 方法 | 読み込まれるタイミング | コンパクション時 | コンテキストコスト | 使うべき場面 |
|---|---|---|---|---|
| CLAUDE.md（ルート） | セッション開始時。全体を通じて残る | 再読み込みされる | 高（全行が常に消費） | ビルドコマンド・構成・コーディング規約・チームの慣習 |
| CLAUDE.md（サブディレクトリ） | 配下のファイルに触れたとき | 再び触れるまで失われる | 低 | サブディレクトリ固有の規約 |
| ルール | 開始時（ユーザーレベル）／paths 指定ならマッチ時のみ | 再注入される | 中（paths 指定で削減） | 特定の制約（例: 全 API ハンドラは入力検証） |
| スキル | 名前と説明のみ開始時。本体は呼び出し時 | 呼び出し済み分を予算内で再注入 | 低 | 手続き的ワークフロー（デプロイ・チェックリスト） |
| サブエージェント | 名前・説明・ツール一覧のみ開始時。本体は Agent 呼び出し時 | 最終メッセージのみ親に返る | 低（独立コンテキストで実行） | 並列作業・隔離して要約だけ返す副次タスク |
| フック | ライフサイクルイベントで発火 | コンパクションを完全にスキップ | 低（設定はコンテキスト外） | 決定的な自動化（リンター・通知・コマンドブロック） |
| 出力スタイル | 開始時にシステムプロンプトへ注入 | コンパクションされない | 高（デフォルトを上書き） | 大きな役割変更 |
| システムプロンプト追記 | 起動時に CLI フラグで渡す | コンパクションされない | 中（初回以降キャッシュ） | トーン・返答の長さ・フォーマットの好み |

### 各方法の要点

**CLAUDE.md** — ルートに置くと常時読み込み。共有リポジトリでは「各チームが追記し、何も削除されない」育ち方をしがちで、すべての行が全員の全セッションでトークンを消費し、**本当に重要な指示への遵守度を薄める**。肥大化したら、チーム固有の慣習はパス指定ルールへ、手順はスキルへ追い出す。**200行未満に保ち、オーナーを決め、コード変更と同じようにレビューする**。「索引」のようなものだと考えるとよい。

**ルール** — `.claude/rules/` のマークダウン。スコープなしなら CLAUDE.md と同じ常時オン。**paths フィールドを付ければ、関係あるファイルに触れたときだけ読み込まれる**。「マイグレーションは追記のみ」のようなファイル固有の制約に最適。

**スキル** — `.claude/skills/` のフォルダ。開始時に読み込まれるのは名前と説明だけで、本文はスラッシュコマンドや自動マッチングで呼び出されたときに読み込む。**デプロイ手順・リリースチェックリスト・レビュー手順のような手続きは CLAUDE.md ではなくスキルに置く**。

**サブエージェント** — `.claude/agents/` のマークダウン。専用のまっさらなコンテキストウィンドウで動作し、親の会話に返るのは最終メッセージとメタデータだけ。**この「隔離性」が、スキルではなくサブエージェントを選ぶ主な理由**。深い調査・ログ解析・依存関係監査のように、参照し直さない中間結果で会話を散らかす副次タスクに使う。手順をメインスレッドで見ながら操縦したいならスキルを使う。

**フック** — ファイル編集・ツール呼び出し・セッション開始などのイベントで発火する、ユーザー定義のコマンド／HTTP／LLM プロンプト。**すべて決定的にトリガーされる**（command・HTTP・mcp_tool は実行も決定的）。設定がコンテキスト外にあるためコストが低い。編集後のリンター実行・完了時の通知・特定コマンドの実行前ブロックなど、**確実に起きてほしいことに使う**。

**出力スタイル** — システムプロンプトに注入され、最も強い指示遵守の重みを持つ。ただしデフォルトの重要な指示（検証の習慣・変更範囲の決め方など）を置き換えてしまうため慎重に。まず組み込みの Proactive / Explanatory / Learning を確認する。

**システムプロンプトへの追記** — 出力スタイルと違い、デフォルトの役割に**加算的にのみ**働く。特定のコーディング標準・出力フォーマット・ドメイン知識の追加に向く。与える指示が多いほど（特に矛盾する場合）遵守度は下がる。

### 「置き場所を見直すべきサイン」

- **「毎回 X のときは必ず Y」を CLAUDE.md に書いている** → フックへ。モデルが自分の判断でフォーマッタを実行することと、フォーマッタが自動実行されることは別物
- **「これは絶対にしてはいけない」を CLAUDE.md に書いている** → 指示という手段が不適切。長いセッション・曖昧な状況・プロンプトインジェクションでは従えないことがある。本当のガードレールは決定的でなければならず、実現手段は**フック（PreToolUse で検査し、コード2で終了して拒否）と権限設定**。組織全体に及ぶ強制は管理者デプロイの管理設定で
- **CLAUDE.md に30行の手順を書いている** → 手順はスキルへ。CLAUDE.md は常に保持すべき「事実」のためのもの
- **paths を指定しない API 固有のルールを書いている** → paths でスコープを絞る。スコープのないルールは CLAUDE.md に書くのと機械的に同じ
