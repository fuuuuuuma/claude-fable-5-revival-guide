設定修復プロンプト（memory-dream）

CLAUDE.md / Skills / メモリに溜まったノイズ——期限切れの相対日付・削除済み手順への参照・重複・矛盾——を一括で整理するプロンプトです。Fable 5 のような上位モデルへ切り替えたタイミングで実行すると、広い文脈での矛盾検出が期待できます。

  • そのまま貼るだけで動く（設定不要）
  • 繰り返し使うなら .claude/skills/memory-dream/SKILL.md として保存しておくと一言で呼び出せる
  • 出力は自分でレビューしてから確定（勝手に確定させない）
PROMPT ① 
memory-dream を実行して、現在の記憶階層全体（CLAUDE.md / Skills / メモリファイル）を整理してください。

手順:
1. Mine: 直近のセッション記録から、繰り返し出た指摘・確定方針・新事実だけを抽出（1回限りのメモは無視）
2. Consolidate: 既存ファイルへマージ。相対日付→絶対日付、矛盾は最新で上書き、死んだ参照は削除
3. Dedup & Resolve: 上位ファイルにあるルールを下位から削除。メタ説明は残さない
4. Prune & Index: 索引は200行以内に。ノート一覧を実ファイルから再生成

鉄則: 経緯・失敗談・履歴は書かない。現時点で正しい知見・ルール・再現手順のみ残す。
出力はレビューできる形で、変更をコミットとして提示してください。
⬇ ファイルでダウンロード（.md）

ハーネス構築プロンプト

リポジトリの Claude Code 設定（ハーネス＝CLAUDE.md・ルール・スキル・フック・検証の仕組み）を10の基準で診断し、「診断 → 一覧表 → 承認 → 移行」の順で再設計させるプロンプトです。いきなり書き換えさせず、人間のレビューを挟む設計にしています。

PROMPT ② 
このリポジトリの Claude Code 設定（ハーネス）を、公式ベストプラクティスに沿って診断し、
再設計案を提示してください。基準は以下です。

【置き場所の基準】
1. CLAUDE.md（ルート）は「常に必要な事実」（ビルドコマンド・構成・規約）だけに絞り、200行未満に保つ
2. 「毎回Xのときは必ずY」型の指示は、CLAUDE.md から settings.json のフックへ移す
3. 「絶対にしてはいけない」型の禁止は、フック（PreToolUse で拒否）と権限設定で強制する
4. 30行を超える手順は .claude/skills/ のスキルへ移す
5. 特定パスにしか関係ない規約は、paths 指定つきのルール（.claude/rules/）へ移す
6. 個人の好みはユーザーレベルの設定ファイルへ移す
7. 深い調査・ログ解析のような隔離できる副次タスクは、サブエージェントとして定義する

【ループの基準】
8. 繰り返し実行するタスクには「完了の定義」と「試行回数の上限」を明文化する
9. 成果物の自己検証手順（テスト・チェックリスト・ルーブリック）をスキルとして書き出す
10. 実装役とは別に、レビュー役（検証者）のエージェント運用を用意する

手順: まず現状の CLAUDE.md / rules / skills / hooks / agents を読み、
「どの指示を・どこへ・なぜ動かすか」の一覧表を出してください。
私が承認したら、実際の移行を1項目ずつ実行してください。
⬇ ファイルでダウンロード（.md）

公式ブログ3本の日本語まとめ

ハーネス構築プロンプトの背景にある考え方を、公式ブログ3本ぶんの日本語まとめとして1ファイルに収録しました。通読すると「なぜその基準なのか」まで分かる教科書パートです。

  • ループ入門（Getting started with loops）— 4種類のループと「止め方」の設計
  • エージェントチーム（Building effective human-agent teams）— 役割・北極星・Doer-Verifier
  • 指示の7つの置き場所（Steering Claude Code）— CLAUDE.md／ルール／スキル／フックの使い分け表
